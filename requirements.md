# 杭州置换买房信息系统（需求文档）

> 视角：我（用户）用于买房决策与信息沉淀  
> 当前优先：**小区信息 + 可视化对比（Phase 1）**  
> 说明：本系统会迭代，字段/展示/流程未来可能调整。

---

## 1. 背景

我和父母计划在杭州进行房产置换。我需要一个轻量的信息系统，用来把看房过程中分散的信息（小区特征、交通、物业、配套、房产形态等）沉淀下来，并以可视化、可筛选、可对比的方式呈现，从而支撑 shortlist 与最终决策。

---

## 2. 总体目标

1. **信息沉淀（可编辑、可追溯）**  
   - 小区/房源/看房记录等信息可持续记录与维护  
   - 数据格式简单，便于手工编辑与后续自动化写入  
   - Git 版本化，保留“当时为什么这么判断”的历史

2. **展示与决策支持（可视化、可对比）**  
   - 可视化展示各小区/房源的关键特征  
   - 支持筛选、排序、多选对比  
   - 移动端优先（看房路上可用）

3. **可扩展（未来逐步演进）**  
   - Phase 1 只做小区  
   - Phase 2 增加房源展示  
   - Phase 3 增加自动抓取/更新（可选）

---

## 3. 非功能要求（NFR）

- **移动端优先**：响应式布局；对比表可用（可横向滚动）；交互不依赖鼠标 hover。
- **组件成熟**：优先使用成熟 UI 组件库，减少手搓样式与兼容问题。
- **静态部署**：站点为静态页面，托管在 GitHub Pages。
- **自动构建**：push 到 main 后自动 build & deploy。
- **仓库可见性**：仓库先 public（未来如需“简单密码访问”，另行方案评估）。
- **隐私与合规**：避免写入敏感个人信息（精确门牌、电话、姓名等）。

---

## 4. 数据与存储约束

- **小区数量**：5–6 个，优先手写维护。
- **数据格式**：YAML。
- **图片数量**：少量，全部放 GitHub 仓库内，以相对路径引用。
- **距离口径**：统一记录为 **步行导航距离（米）**，网页侧可换算步行/骑行时间。
- **关注面积段**：固定三档  
  - 80–90㎡  
  - 90–100㎡  
  - 100–120㎡

---

## 5. 阶段规划（可调整）

### Phase 1（当前优先：先做小区）
- 小区 YAML 数据录入与展示  
- 小区列表/详情/对比页  
- tag、多维筛选、排序  
- 勾选对比（2–6 个）  
- 图表：基于现有字段的数值对比（不引入评分体系）  
- GitHub Actions 自动 build & deploy  
- 移动端优先（responsive）

### Phase 2（房源展示）
- 房源结构化数据与展示（与小区关联）  
- 房源列表筛选/排序/详情  
- 基本变更记录（可选）

### Phase 3（自动抓取/更新）
- shortlist 房源的变更检测（上下架、调价等）  
- 定时任务运行（GitHub Actions cron 或其他）  
- 变更历史可视化（可选）

---

## 6. 可能变化点（允许未来调整）

- 小区字段可能补充/删减（尤其是物业/交通/房产形态细节）。
- 图表维度、默认排序、筛选逻辑可能调整。
- 若引入访问控制（网页密码/登录），可能需要调整托管方案（GitHub Pages 静态站不适合做真正安全的登录）。
- 房源模块引入后，信息架构与页面层级会扩展。

---

## 7. 成功标准（Phase 1）

- 以 YAML 维护小区信息，改动可追溯，新增/修改后网站自动更新。
- 列表页可筛选（tag 等）与排序，移动端体验良好。
- 可勾选 2–6 个小区进行对比，得到清晰的对比表与基础图表。
- 数据校验有效：字段不合法会阻止构建，避免线上展示错误数据。
